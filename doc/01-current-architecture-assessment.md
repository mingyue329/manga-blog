# 当前项目架构评估

## 1. 项目现状

当前项目是一个以 Vite + React + React Router 为基础的前端静态博客，页面结构清晰，适合继续演化成公开模板或内容型站点。

现有架构特点：

- 路由集中在 `src/app/router.tsx`
- 页面通过 loader 组装数据
- 内容主源是本地 Markdown
- 页面内容服务位于 `src/features/*/*-service.ts`
- 全站布局、头部、页脚、主题样式已初步抽离
- 当前没有后端、数据库、CMS、鉴权体系

从代码组织上看，它已经不是“单页堆代码”，而是具备了继续扩展的基础。

## 2. 现有优点

### 2.1 内容与页面已经有初步分层

例如文章列表页和详情页都不是直接在页面组件中硬编码内容，而是通过 service 汇总后再喂给页面：

- `src/features/posts/posts-page-service.ts`
- `src/features/posts/post-detail-page-service.ts`
- `src/features/posts/article-content-service.ts`

这意味着后续替换内容源时，不需要整页重写。

### 2.2 页面按 feature 拆分，结构合理

当前目录按 `home / posts / about / shared` 分区，这种拆分适合继续新增模块，例如：

- `features/steam`
- `features/search`
- `features/feed`
- `features/admin`

### 2.3 样式已经具备站点主题感

`src/index.css` 和 `shared/ui` 已经形成统一视觉语言。对公开模板来说，这一点很重要，因为后续可以把“主题系统”抽成一层稳定能力，而不是每页单改。

### 2.4 静态部署成本低

当前架构天然适合：

- GitHub Pages
- Cloudflare Pages
- Vercel 静态输出
- Netlify

这让它非常适合作为“公开博客源码”的第一版。

## 3. 当前不足

### 3.1 内容模型还没有正式标准化

虽然文章内容已经来自 Markdown，但目前更偏“项目内部可用”，还不够“模板化”。后续需要明确：

- frontmatter 字段标准
- slug 生成规则
- 草稿字段
- 发布时间字段
- 推荐 / 置顶字段
- 封面图字段
- 标签和分类字段

如果不先定模型，后面接 Steam、RSS、SEO、搜索时都会反复返工。

### 3.2 数据源适配层还不够明确

现在 service 层虽然已经存在，但还没有形成显式的数据源接口。当前更像“本地内容服务”，不是“可替换内容源”。

建议后续抽出统一接口，例如：

- `BlogContentSource`
- `SteamSource`

这样页面层只依赖接口，不依赖本地 Markdown 或未来 API 的具体实现。

### 3.3 SEO 能力还不完整

公开博客模板必须补齐：

- 页面级 metadata
- Open Graph
- Twitter Card
- canonical
- sitemap
- robots.txt
- RSS

否则它更像一个“前端展示站”，不是完整博客产品。

### 3.4 工程文档不足

如果准备开源，目前还缺：

- 如何新增文章
- 如何替换站点品牌
- 如何修改导航
- 如何替换首页模块
- 如何部署
- 如何配置图片资源

没有这些，别人拿到仓库以后依然很难复用。

### 3.5 缺少可持续扩展的边界设计

当前项目还没有明确区分：

- 纯静态能力
- 需要服务端的能力
- 内容模型
- 第三方集成模型

这会导致未来一旦加 Steam、评论、订阅、搜索，就容易把现有结构拉乱。

## 4. 架构判断

这个项目目前最适合被定义为：

“基于本地 Markdown 内容源的前端静态博客模板，已具备主题化和模块化雏形，但尚未完成模板化、SEO 化和数据源抽象。”

这个判断很重要，因为它决定了后续不要直接冲着“全功能 CMS”去改，而应该先把它打磨成一个稳定的静态博客核心。

## 5. 推荐的演进方向

### 第一阶段：把它整理成稳定的静态博客核心

目标：

- 统一站点配置
- 标准化文章模型
- 稳定文章列表与详情页能力
- 补齐 SEO 和内容输出能力
- 让别人拿到仓库就能跑、能改、能发

### 第二阶段：补数据源抽象

目标：

- 页面层不再直接依赖本地内容实现
- 为 Steam、本地 JSON、未来 CMS 预留接入面

### 第三阶段：再决定是否引入后端

目标：

- 只有当功能确实需要动态、缓存、鉴权、定时同步时，再上后端
- 不为了“看起来完整”提前引入复杂度

## 6. 立即建议

当前优先级建议如下：

1. 抽站点配置和内容模型
2. 补开源模板文档和示例内容
3. 补 SEO、RSS、sitemap
4. 设计 Steam 数据结构，但先不接后端
5. 等 Steam 功能确定后，再评估是否增加 BFF / API 层

## 7. 总结

当前架构方向是对的，最大的问题不是“技术选型错了”，而是“产品边界和扩展边界还没定清楚”。

正确路线不是推倒重来，而是：

- 先把静态博客核心做稳
- 再把内容和第三方集成抽象清楚
- 最后按需求引入后端

这样扩展 Steam、公开源码、后续接 CMS 或 API 都会比较顺。
