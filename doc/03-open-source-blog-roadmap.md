# 公开博客源码改造清单

## 1. 目标定义

如果这个项目未来要作为公开博客源码发布，它不能只是“你自己能跑”的站点，而应该满足这几个条件：

- 别人克隆后能快速启动
- 别人能读懂怎么加内容、改配置、换主题
- 别人能在不理解全部源码的情况下完成基础定制
- 项目具备稳定的内容模型和扩展边界

从这个标准看，当前项目已经有不错的视觉完成度，但还缺少模板化所需的工程化补强。

## 2. 必补能力

### 2.1 站点配置中心

目前已经有部分站点配置，但还需要进一步收敛为统一入口，至少包括：

- 站点标题
- 站点副标题
- SEO 描述
- 导航
- 页脚
- 社交链接
- 品牌文案
- 首页 hero 文案

建议形成：

- `src/shared/site/site-config.ts`
- `src/shared/site/site-metadata.ts`
- `src/shared/site/site-theme.ts`

这样别人换站点信息时，不用改多个 feature 文件。

### 2.2 标准化文章模型

建议明确 frontmatter 或文章元数据规范，至少包含：

- `title`
- `slug`
- `excerpt`
- `date`
- `tags`
- `category`
- `cover`
- `coverRatio`
- `featured`
- `draft`
- `series`
- `relatedGames`

这一步做完后，文章源才能真正稳定。

### 2.3 内容录入流程

公开模板一定要把“怎么发文章”做简单。

建议补：

- 新文章模板文件
- frontmatter 示例
- 图片目录规范
- slug 命名规范
- 封面图比例规范
- 标签命名规范

最好在 `README.md` 里给出最小示例。

### 2.4 SEO 与内容输出

博客模板几乎都应该有：

- `sitemap.xml`
- `robots.txt`
- RSS feed
- 页面级 metadata
- Open Graph 图片策略
- canonical URL

当前这部分是明显短板。

### 2.5 示例数据与占位资源

公开模板不能只放高度个人化内容。需要准备：

- 示例首页文案
- 示例关于页内容
- 示例文章
- 示例封面图
- 示例标签与分类

也建议把过于个人化的文本替换为更通用的默认值。

## 3. 建议新增的工程结构

为了后续扩展，建议逐步补下面这些层：

### 3.1 内容源目录

建议单独收口：

- `src/content/posts`
- `src/content/site`
- `src/content/steam`

不要让内容和页面实现混在一起。

### 3.2 数据访问层

建议统一为：

- `src/features/posts/post-repository.ts`
- `src/features/steam/steam-repository.ts`
- `src/shared/content/content-source.ts`

这样未来切换到 API / CMS 时，不需要改 UI 组件。

### 3.3 类型层

当前类型已经在 `src/shared/types/content.ts` 中集中了一部分，后面可以继续拆成：

- `site`
- `post`
- `steam`
- `seo`

避免一个文件越堆越大。

## 4. 公开源码前建议补齐的文档

至少建议新增以下文档：

- `README.md`
- `doc/01-current-architecture-assessment.md`
- `doc/02-steam-integration-options.md`
- `doc/03-open-source-blog-roadmap.md`
- `doc/content-model.md`
- `doc/deployment.md`
- `doc/customization-guide.md`

其中最关键的是：

- 如何启动
- 如何新增文章
- 如何换品牌
- 如何部署
- 如何扩展模块

## 5. 需要清理的风险点

### 5.1 个人化内容过重

如果要开源，建议逐步把下面这些内容改成通用可替换：

- 个人品牌文案
- 高度私人化页面内容
- 过强绑定个人身份的字段
- 临时样例文本

开源模板应该给“替换入口”，而不是让别人到处搜字符串。

### 5.2 注释和编码一致性

当前仓库里曾出现过注释乱码痕迹。开源前需要统一：

- 文件编码
- 注释语言
- 注释风格

否则阅读体验会明显掉档。

### 5.3 构建与部署说明不足

需要写清：

- Node / pnpm 版本要求
- 如何本地开发
- 如何构建
- 如何部署到静态托管
- 基础路径 `basename` 的使用方式

### 5.4 内容与样式耦合点

当前部分页面文案和视觉结构绑定较深。未来如果别人替换内容后版式容易崩，就说明模板抽象还不够。

应尽量把：

- 文案
- 模块开关
- 卡片列表数据
- 主题 token

从页面实现里抽出来。

## 6. 推荐的里程碑

### 里程碑 A：整理成稳定可维护的个人博客

目标：

- 站点配置统一
- 文章模型统一
- 页面样式层级稳定
- SEO 初步补齐

### 里程碑 B：整理成可公开复用模板

目标：

- README 完整
- 示例内容齐全
- 模块结构清晰
- 定制入口明确
- 内容目录和主题目录抽离

### 里程碑 C：扩展成可插拔内容平台

目标：

- 支持本地 Markdown
- 支持 Steam 数据源
- 支持未来 API / CMS
- 保持页面组件不感知底层来源

## 7. 推荐优先级

如果按投入产出比排序，我建议先做：

1. 统一站点配置和内容模型
2. 清理个人强绑定内容
3. 补 README 和新增文章流程
4. 补 SEO、RSS、sitemap
5. 抽内容源接口
6. 设计 Steam 模块
7. 再考虑后端

## 8. 一个更稳的公开路线

比较合理的发布方式不是一次性把所有能力都做完，而是分两版：

### v1

定位为：

“可静态部署的漫画感博客模板”

包含：

- 首页
- 文章列表
- 文章详情
- 关于页
- Markdown 内容支持
- 站点配置

### v2

定位为：

“支持游戏内容模块扩展的博客模板”

包含：

- Steam 模块
- 更完整的内容模型
- 更完整的 SEO
- 数据源接口抽象

## 9. 总结

如果要把这个项目做成公开博客源码，重点不是“多做页面”，而是补齐这三件事：

- 模型稳定
- 配置清晰
- 扩展边界明确

只有这三件事做稳了，它才会从“一个做得不错的个人站”变成“别人愿意拿来复用的源码模板”。
